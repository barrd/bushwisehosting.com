BWH - Site Conventions

CSS Controlled Media Types…

Everything mentioned is of course highly dependent on whether your Browser/UA of choice can handle (or interpret) the code used. We test as extensively as we can but if a Browser/UA is either broken, bugged or incapable of handling the data we apologise but there's not a lot we can do (at present) that we have not as yet implemented.

Print Conventions:

When printing any BushWise page the Print Style Sheet determines how the page should be displayed (and ultimately printed). This includes removing the Page Header, Sidebar & Related navigation panes, whilst adding a small greyscale BushWise logo to the top of the printable document for identification.

HyperText links are not visually indicated as such (not much use on a printed page anyway) and where applicable (or possible), text indicating a link is removed (not shown) or alternative text applied.

Handheld Conventions:

Web enabled Personal Digital Assistant's, Phones etc:

One word… “tricky”. Just when we all thought that catering for the plethora of screen based Browsers was a challenge, handheld devices quite simply come into a league of their own. There are no “standards” to speak of regarding just how any of these devices will render (or rather “attempt” to render) any given data supplied to them. The big issue of course being what is known as “real estate” - the amount of screen space available for viewing data flowed into your handheld device of choice. This variable changes quite dramatically and causes developers a bit of a headache. We have gone for the “keep it small, keep it simple” approach which seems to work for most scenarios, however some layout devices such as HTML table's may, or may not render as visually understandable or indeed “at all”.

Wireless Application Protocol - WAP

A decision was made not to go down this particular road, it seems a dying (if not already dead) means of supporting handheld devices and utilising CSS seems a far more practical and ultimately useful tool to maintain a service for these devices.

Aural & Tactile Feedback:

An area we are very interested in expanding, whilst quite a bit of research has been done we do not yet have any systems in place to incorporate the appropriate media required as we currently have no access to the hardware devices or software. It would be a great help to communicate to someone (or persons) who have access to, are familiar with and are willing to act as a “guinea pig” whilst we add the required elements needed to enable access for those with visual difficulties.

Web Standards:

The World Wide Web Consortium - W3C

The following documents have been used as the reference materials in the creation of this Site:

  • Cascading Style Sheets, level 2 revision 1 CSS 2.1 Specification
  • HTML 4.01 Specification
  • XHTMLTM 1.0 The eXtensible HyperText Markup Language (Second Edition)
  • Web Content Accessibility Guidelines 1.0

This Site fully 100% conforms to the standards of the first three (CSS, HTML & XHTML) and follows as closely as possible the last (Accessibility Guidelines).

Erratum…

Site Acronyms & Abbreviations

Many content authors are confused as to how the acronym element should be used. XHTML 2 will use the abbr element to represent all types of abbreviations, including acronyms.
~ xhtml.com

The acronym tag is to be made redundant (deprecated) and to be incorporated under the abbr tag according to the current documentation surrounding XHTML 2.
